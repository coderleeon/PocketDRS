# PocketDRS

## Overview

PocketDRS is a browser-based cricket analytics platform designed for home, indoor, and society cricket.

The platform provides a low-cost DRS-style experience using a single smartphone camera and a fixed playing environment.

The initial target environment is a tiled indoor pitch where the wicket is represented using a maroon cardboard marker.

The long-term vision is to provide:

* LBW Detection
* Ball Tracking
* Bowling Speed Estimation
* Pitch Maps
* Bowling Analytics
* DRS Replays

without requiring expensive hardware.

---

# Core Philosophy

Build the smallest working product first.

Priority order:

1. Calibration
2. Overlay Rendering
3. Ball Tracking
4. LBW Detection
5. Speed Detection
6. Analytics
7. DRS Replay

Reliability is more important than advanced AI features.

# Critical Architecture Requirement

The browser is the primary computer vision engine.

The backend is primarily a decision and analytics engine.

Recorded videos should remain on the user's device whenever possible.

Do not upload videos by default.

---

# Processing Pipeline

## Browser

The browser performs:

* Camera access
* Video recording
* Calibration
* Overlay rendering
* Ball detection
* Ball tracking
* Coordinate extraction

The browser should generate lightweight structured cricket data.

Example:

{
"fps": 60,
"ball_positions": [
[412, 920],
[418, 901],
[426, 882]
],
"impact_frame": 47
}

Only this data should be sent to the backend.

---

## Backend

The backend receives:

* Ball coordinates
* Bounce coordinates
* Impact coordinates
* Calibration information
* Match metadata

The backend returns:

* LBW result
* Predicted trajectory
* Bowling speed
* Analytics

---

# Playing Environment

Assumptions:

* Fixed camera position
* Indoor tiled pitch
* Consistent lighting
* Maroon cardboard wicket marker

Only the center region of the cardboard represents the wicket.

The outer cardboard boundaries should not be treated as stumps.

---

# Calibration System

Users perform calibration once.

The system should store:

* Pitch boundaries
* Crease coordinates
* Wicket coordinates
* Homography matrix

Calibration should persist across sessions.

The user should not need to recalibrate every match unless the camera position changes.

# MVP Success Criteria

Version 0 is considered successful if a user can:

1. Open the website.
2. Calibrate the pitch.
3. Save calibration.
4. See a live pitch overlay.
5. Record a delivery.
6. Replay the recording.

No LBW detection is required for Version 0 completion.

The first milestone is proving accurate calibration and overlay rendering on mobile browsers.
# Future Feature: AR-Assisted Calibration

Version: Future (Post-MVP)

Objective:

Reduce manual calibration effort using camera-based scene understanding.

Workflow:

1. User opens calibration mode.
2. User slowly scans the tiled pitch area.
3. System detects:

   * Floor plane
   * Tile boundaries
   * Wicket marker
   * Pitch orientation
4. System automatically suggests:

   * Pitch boundaries
   * Crease positions
   * Wicket location
5. User reviews and adjusts if necessary.
6. Calibration profile is saved.

Important:

AR-assisted calibration is optional.

Manual calibration must always remain available as a fallback.

Target Experience:

Similar to the setup experience of mobile measurement applications where the camera scans the environment and proposes geometry automatically before user confirmation.
